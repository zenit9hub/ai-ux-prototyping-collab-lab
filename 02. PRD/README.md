# 02. PRD

## 이 단계의 역할

여기는 리서치·요구사항 후보를 **제품 의사결정 가능한 문서**로 바꾸는 단계다. 무엇을 왜 만들고, 누구를 위해, 어디까지 만들며, 무엇으로 완료를 판단할지를 명확히 한다.

## 입력 기준

- `01. 요구사항 수집·리서치`의 accepted evidence map과 요구사항 후보
- 과제의 사업·운영·기술 제약
- `00`의 공통 guardrail과 변경 관리 규칙

입력 리서치에 빈 부분이 있으면 PRD의 사실처럼 덮지 않는다. ASSUMPTION 또는 OPEN으로 남기고 결정자를 지정한다.

## 핵심 산출물

| 파일 예시 | 내용 | 다음 수신자 |
| --- | --- | --- |
| `<CASE-ID> — PRD.md` | 문제, 사용자, 목표, 범위/비범위, 요구사항, 우선순위, acceptance criteria | UX / IA Owner |
| `<CASE-ID> — 요구사항 Traceability.md` | `R-##` ID와 근거·결정·수용 기준의 연결 | UX / IA Owner |
| `<CASE-ID> — PRD 시각 요약.html` | 목표·범위·우선순위·open question의 공유용 보조 자료 | UX / IA Owner |

상세 형식은 [`01. 단계 산출물 템플릿`](<../00. 운영·공통 기준/templates/01. 단계 산출물 템플릿.md>)을 사용한다.

## PRD 필수 질문

1. 사용자가 해결하려는 핵심 문제는 무엇인가?
2. 이번 범위와 명시적 비범위는 무엇인가?
3. 가장 먼저 확인할 성공 기준은 무엇인가?
4. 각 요구사항은 누구에게 어떤 가치를 주며, 어떻게 수용되는가?
5. 다음 IA 단계가 해석해야 할 OPEN은 무엇인가?

## 하지 않는 일

- 메뉴 구조, 화면 수, 시각 스타일을 먼저 고정하지 않는다.
- 불확실한 기능을 “향후 고려”라는 말로 범위에 숨기지 않는다.
- AI가 만든 기능 목록을 검증 없이 요구사항으로 채택하지 않는다.

## 품질 게이트: PRD → IA

- [ ] 문제·대상 사용자·목표가 명확하다.
- [ ] 범위와 비범위가 구분된다.
- [ ] 모든 핵심 요구사항에 `R-##` ID, 우선순위, 수용 기준이 있다.
- [ ] FACT/ASSUMPTION/DECISION/OPEN이 분리된다.
- [ ] UX / IA Owner가 이 문서만으로 정보 구조·흐름·상태를 시작할 수 있다.

## 다음 인수인계

`03. IA`에는 accepted PRD, 요구사항 traceability, 우선순위와 비범위, acceptance criteria, OPEN 목록을 전달한다. UI 화면을 미리 답으로 제시하지 않고, IA가 구조를 탐색할 여지를 남긴다.

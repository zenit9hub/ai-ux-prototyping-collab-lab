# 강의 기준자료와 현업 전환 안내

## 1. 두 작업 공간의 관계

강의 진행 중에는 Obsidian Vault의 이론·실습 자료를 주 참고한다. 그러나 이 저장소는 그 자료를 복제하는 곳이 아니라, 수강생이 **AI Native workflow를 실제 업무 흐름처럼 수행하는 곳**이다.

| 질문 | 강의 기준자료 | 이 저장소 |
| --- | --- | --- |
| 무엇을 배우는가? | AI 에이전트, UX Validation Loop, 하네스, 디자인 프로토타이핑 개념 | 그 개념을 GitHub 협업·인수인계·변경 관리로 실행 |
| 무엇을 만드는가? | 예시·설명·강의용 샘플 | 사례별 근거, PRD, IA, Wireframe, Hi-Fi, handoff |
| 무엇이 버전 기준인가? | 강사가 배포한 수업 자료 | `main`의 accepted baseline과 canonical manifest |
| 수업 후에도 쓰는가? | 참고용 | clone하여 현업 사례에 재사용 |

## 2. 수업 중 사용 방식

1. 강사는 Vault의 이론·실습 자료를 통해 각 단계의 목적과 예시를 안내한다.
2. 수강생은 이 저장소의 `00` 문서와 단계 README를 읽고, Git branch에서 실제 산출물을 만든다.
3. AI 도구는 정해진 input·constraint·evaluation 기준을 받아 초안·변형안을 만든다.
4. 동료 또는 다음 단계 담당자가 PR에서 수신 가능 여부를 검토한다.
5. `accepted`된 산출물만 다음 단계의 입력이 된다.

## 3. 수업 후 현업 전환 방식

수강생은 이 저장소를 새 사례용으로 clone한 뒤 다음만 바꾸면 된다.

- `case_id`와 프로젝트 context
- 실제 리서치 근거와 조직의 제약
- 담당 역할과 reviewer
- 디자인 도구의 canonical source
- 개발팀의 수신 방식과 보안·법무·브랜드 규칙

단계 구조, AI 작업 계약, quality gate, PR checklist, handoff manifest는 유지한다.

## 4. 현업 전환 시 반드시 추가할 것

- 개인정보·고객 정보·보안 분류 기준
- 실제 의사결정자와 RBAC/승인 흐름
- 브랜드·디자인 시스템·접근성 기준
- 실제 개발 저장소, issue tracker, 배포 환경과의 연결
- 조직의 문서 보존·감사·라이선스 정책

## 5. 이 저장소가 독립적으로 유지되어야 하는 이유

강의 Vault의 개인 경로·로컬 도구·특정 수업 사례에 의존하면 수강 후 재사용이 어려워진다. 이 저장소는 필요한 운영 원칙과 template을 자체 포함하고, 외부 자료는 URL·버전·출처로만 참조한다. 따라서 팀·조직·사례가 바뀌어도 작업 방식은 유지할 수 있다.

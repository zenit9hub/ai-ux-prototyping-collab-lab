# 시니어 관점 Q&A·의사결정 기록 — AIoT 예시

역설계 문서는 화면에 보이는 사실을 잘 보존하지만, 그것만으로 실제 업무 규칙까지 확정되지는 않는다. 이 문서는 AI가 만든 결과물을 더 빨리 채우기 위한 질문 목록이 아니라, **확인된 사실·교육용 가정·실서비스 결정 필요 사항을 분리하고, 운영상 중요한 모호함을 질문 → 답변 → 후속 산출물 갱신으로 전환하기 위한 기록**이다. 시니어 담당자는 “무엇을 더 만들까요?”보다 “이 선택이 누구에게 어떤 위험·책임·검증 기준을 만들까요?”를 먼저 묻는다.

> [!warning] 공유용 참고 문서
> 이 문서는 AIoT 교육 사례를 기반으로 한 `reference_only: true` 예시다. 어떤 사례의 accepted baseline이나 실제 운영 정책도 아니다. 수강생은 이 문서의 답을 복사해 확정하지 말고, 자기 사례에서 근거·책임자·결정 기한을 다시 기록해야 한다.

## 이 문서를 쓰는 위치

1. 사례의 `01. 요구사항 수집·리서치`에서 사실과 질문을 수집한다.
2. PRD·IA·Wireframe을 만들며, 다음 단계의 해석을 바꿀 모호함을 이 형식으로 질문한다.
3. 답변이 확인되면 각 단계 문서에 `DECISION`으로 반영하고, 아직 확인되지 않았으면 `ASSUMPTION` 또는 `OPEN`으로 남긴다.
4. 위험한 변경은 [변경 요청(CR) 규칙](<03. AI 협업·변경 관리 규칙.md#4-변경-요청cr-규칙>)에 따라 영향 단계를 추적한다.

## 먼저 잡아야 할 핵심 인사이트

1. **클릭 가능함은 실행 가능함의 증거가 아니다.** command, campaign, export, settings의 UI 흐름은 실제 API·권한·저장·감사가 구현됐다는 뜻이 아니다.
2. **숫자보다 범위가 먼저다.** KPI·위험도·알림·리포트는 모두 “어느 사이트·그룹·기간·시간대의 값인가”가 먼저 설명돼야 한다.
3. **위험 행동은 모달 하나로 안전해지지 않는다.** 실행 권한, 대상 적격성, 영향, 실패 시 다음 행동, 감사 기록이 함께 결정돼야 한다.
4. **빈칸을 숨기지 않는 전달이 좋은 전달이다.** 화면·상태·예외는 구체적으로 넘기되, API·RBAC·계산식처럼 미확정인 항목은 `OPEN`으로 남겨 담당자와 결정 시점을 붙인다.
5. **답변은 문서의 끝이 아니라 다음 입력이다.** 한 질문의 답은 PRD, IA, Wireframe, Hi-Fi, Handoff 중 적절한 문서를 갱신하고 검증 시나리오로 다시 확인한다.

## 우선 확정 Q&A — 운영 판단을 먼저 맞춘다

### QA-01. 이번에 만들고 검증할 범위는 어디까지인가?

**왜 지금 묻는가**
화면이 그럴듯할수록 협업자는 이를 실제 운영 기능으로 오해하기 쉽다. 먼저 “이번 결과물이 무엇을 증명하고 무엇을 증명하지 않는지”를 고정해야 이후의 대화가 과장되지 않는다.

**FACT — AIoT 예시에서 확인한 것**
7개 주 화면과 5개 오버레이, 4개 사이트·4개 그룹·16개 장비의 시나리오, alert·command·campaign의 상태 변화가 있다. 반면 API, 로그인, 실제 권한 검증, 데이터 저장, 실기기 명령, 실제 export는 확인되지 않았다.

**ASSUMPTION — 교육용 합의안**
이번 prototype은 `위험 탐지 → 조사 → 안전한 조치 시연 → 결과 확인`의 UI·상태 흐름을 검증한다. 예시 데이터와 시연 상태는 화면에서 드러내며, 실제 운영 기능처럼 약속하지 않는다.

**OPEN — 실제 사례에서 확인할 것**
인증 방식, API, 실시간 telemetry, 실제 장비 연동, RBAC, 감사 로그, export 파일 생성은 제품·개발·보안 책임자가 별도로 결정한다.

**다음 반영**
01의 범위·비범위, 05의 예시 데이터/시연 경계, 06의 미해결 handoff 항목에 반영한다.

> **핵심 인사이트**: prototype의 완성도와 서비스의 완성도를 분리하는 것이, 실무형 AI 활용의 첫 안전장치다.

### QA-02. 대시보드의 ‘현재 범위’는 무엇이며, 모든 모듈이 같은 범위를 쓰는가?

**왜 지금 묻는가**
대시보드 제목은 fleet-wide인데 일부 KPI가 단일 장비 기준으로 계산되거나, 위험 장비·최근 알림이 필터와 다르게 보이면 사용자는 숫자 자체를 신뢰하기 어렵다.

**FACT — AIoT 예시에서 확인한 것**
Dashboard header, KPI, 위험 장비, 최근 알림, 시간 필터의 적용 범위가 완전히 일치하지 않았다. `0건`일 때 전체 fleet로 되돌아가면 안 된다는 보정 필요성도 확인됐다.

**ASSUMPTION — 교육용 합의안**
기본 범위는 `All sites · All groups · Last 24 hours`로 두고, Apply된 site·group·기간은 Header, KPI, Risk, Alerts, Reports에 같은 의미로 반영한다. 전체 fleet 비교가 필요한 모듈은 필터 예외임을 별도 라벨로 밝힌다.

**OPEN — 실제 사례에서 확인할 것**
기간의 기준 시각, site timezone, 실제 데이터 지연 허용치, 집계 기준은 데이터·운영 책임자가 정한다.

**다음 반영**
02의 요구사항·수용 기준, 03의 filter/scope 상태, 04의 scope summary, 07의 범위 검증 시나리오에 반영한다.

> **핵심 인사이트**: 시니어는 KPI를 예쁘게 다듬기 전에 “이 숫자가 무엇을 포함하고 무엇을 제외하는가”를 고정한다.

### QA-03. 위험도·Health·Connectivity는 각각 무엇을 뜻하며, ‘지금 조치 필요’는 무엇으로 판단하는가?

**왜 지금 묻는가**
`Critical` 배지 하나만으로는 사용자가 원인과 행동을 판단할 수 없다. 위험도, 장비 상태, 연결 상태가 같은 의미로 섞이면 잘못된 조치로 이어질 수 있다.

**FACT — AIoT 예시에서 확인한 것**
Device에는 health, connectivity, firmware, signal, battery, last seen, risk score가 함께 보인다. 위험도 계산식, 신선도 기준, 텔레메트리 단위는 확정되지 않았다.

**ASSUMPTION — 교육용 합의안**
Health와 Connectivity는 별도 상태로 유지하고, Risk는 조사 우선순위를 돕는 보조지표로 사용한다. 화면에는 마지막 갱신 시각과 `Unknown / Stale / Unavailable` 상태를 보여 주며, 값이 없을 때 임의의 정상값으로 채우지 않는다.

**OPEN — 실제 사례에서 확인할 것**
위험 점수 계산식, critical 임계치, health 우선순위, signal·battery의 단위와 샘플링 주기는 데이터·도메인 책임자가 확정한다.

**다음 반영**
02의 데이터 신뢰성 요구사항, 03의 상태 모델, 04의 Device Detail annotation, 06의 표시 데이터 계약에 반영한다.

> **핵심 인사이트**: 위험은 ‘빨간색’이 아니라, 사용자가 원인을 추적하고 다음 행동을 선택할 수 있는 설명 가능한 상태여야 한다.

### QA-04. 알림을 ‘해결됨’으로 바꾸는 최소 증거와 책임 이동 규칙은 무엇인가?

**왜 지금 묻는가**
담당자 배정과 해결 처리는 다른 일이다. 상태만 녹색으로 바꾸면 누구도 무엇을 했는지, 왜 끝났는지, 재발 시 누가 이어받는지 알 수 없다.

**FACT — AIoT 예시에서 확인한 것**
Alert는 `Open → In Progress → Resolved` 흐름과 owner 변경을 보인다. 해결 메모, 재오픈, SLA·에스컬레이션, 동시 수정 처리 방식은 미확정이다.

**ASSUMPTION — 교육용 합의안**
`Resolved`에는 최소한 조치 요약 또는 해결 근거를 남긴다. 담당자 배정만으로 해결 상태를 만들지 않으며, 검증 실패·재발이 확인되면 기존 이력을 보존한 채 `Reopen`할 수 있는 상태를 둔다.

**OPEN — 실제 사례에서 확인할 것**
해결 메모/코드의 필수 여부, 재오픈 권한, SLA, 에스컬레이션, 담당자 알림, 동시 편집 충돌 정책은 운영 책임자가 정한다.

**다음 반영**
03의 alert flow/state, 04의 Alert Detail, 06의 alert 상태 전이와 audit 항목에 반영한다.

> **핵심 인사이트**: `Resolved`는 색상 상태가 아니라 책임 종료의 근거와 후속 조치가 남은 운영 기록이다.

### QA-05. 원격 명령은 누가, 어떤 조건에서, 어떤 실패 결과까지 책임지고 실행하는가?

**왜 지금 묻는가**
Restart나 Firmware Update 같은 행동은 디자인 문제를 넘어 운영 중단 위험과 연결된다. “확인 체크박스를 넣었으니 안전하다”는 답으로는 충분하지 않다.

**FACT — AIoT 예시에서 확인한 것**
Command은 확인 → 진행 → 성공/실패 → 재시도 UI를 제공하고, offline 장비는 시연상 실패한다. checkbox는 실제 RBAC가 아닌 시연용 acknowledgement다.

**ASSUMPTION — 교육용 합의안**
명령 실행 전에는 대상 장비, 예상 영향, 현재 상태, 실행자/권한 상태를 명시한다. 실행 결과에는 시작·완료 시각, 실패 사유, 안전한 다음 행동, 재시도 가능 여부를 남긴다. 실제 권한을 구현하지 않은 prototype은 `Authorization pending validation`과 같은 표기로 한계를 숨기지 않는다.

**OPEN — 실제 사례에서 확인할 것**
role/RBAC matrix, 장비 capability, 실행 금지 조건, 승인 단계, timeout, cancellation, idempotency, retry 횟수, audit 보존 기준은 제품·보안·플랫폼 담당의 결정이 필요하다.

**다음 반영**
02의 위험 행동 요구사항, 03의 command 상태, 05의 safety state, 06의 권한·감사·API blocker에 반영한다.

> **핵심 인사이트**: 노련한 질문은 “버튼을 만들 수 있나?”가 아니라 “실행해도 되는가, 실패하면 누가 무엇을 하는가?”에 닿는다.

### QA-06. 펌웨어 배포는 어디까지 진행하고, 실패가 보이면 언제 멈추는가?

**왜 지금 묻는가**
여러 장비에 영향을 주는 배포에서 Create와 Deploy 버튼의 존재만으로는 안전한 흐름이 만들어지지 않는다. 배포 확대와 중단의 판단 기준이 필요하다.

**FACT — AIoT 예시에서 확인한 것**
대상 그룹·버전·장비 선택·승인·장비별 결과·retry가 있고, campaign과 target의 상태가 분리돼 있다. staged rollout, pause/cancel/rollback, 실패 임계치는 확정되지 않았다.

**ASSUMPTION — 교육용 합의안**
교육용 MVP에서도 `대상 적격성 확인 → 소규모 대상 배포 → 장비별 결과 확인 → 실패 시 pause/retry 검토`의 순서를 보여 준다. campaign 전체 상태와 device별 결과를 분리하고, 실패를 성공 요약 뒤로 숨기지 않는다.

**OPEN — 실제 사례에서 확인할 것**
배포 단위, 실패 임계치, 승인자, 자동/수동 rollback, pause·cancel 권한, 재시도 한도는 firmware 운영 정책으로 확정해야 한다.

**다음 반영**
03의 campaign flow/state, 04의 Campaign 화면, 06의 rollout·rollback handoff blocker에 반영한다.

> **핵심 인사이트**: 배포 UX의 본질은 “배포 시작”이 아니라, 이상이 보일 때 확산을 멈추고 판단할 수 있는 가드레일이다.

## 후속 확인 Q&A — 일관성·전달·검증으로 연결한다

### QA-07. 필터는 언제 적용되며, 빈 결과와 화면 이탈은 어떻게 처리하는가?

**왜 지금 묻는가**
사용자는 검색·필터를 바꾼 순간 결과가 바뀌는지, Apply 후 바뀌는지 예측해야 한다. 작은 규칙의 불일치도 데이터가 틀렸다는 인상을 만든다.

**FACT — AIoT 예시에서 확인한 것**
다수의 필터가 draft와 applied 값을 나누며 Apply 후 반영된다. Device Detail에서 돌아올 때 이전 필터·스크롤·선택 상태를 보존하는 규칙도 제안돼 있다.

**ASSUMPTION — 교육용 합의안**
Search의 즉시 반영 여부를 명시하고, 나머지 필터는 `Draft → Apply → Applied` 흐름으로 통일한다. 0건이면 적용 조건을 요약하고 Reset 또는 조건 완화 행동을 제공하며, 전체 fleet으로 조용히 되돌아가지 않는다. Detail에서 Back하면 조사 출처의 applied scope와 문맥을 보존한다.

**OPEN — 실제 사례에서 확인할 것**
서버 검색/클라이언트 검색의 구분, pagination, URL 공유, 임시 draft의 이탈 경고·저장 정책은 구현 방식과 함께 확정한다.

**다음 반영**
03의 navigation/filter state, 04의 filter·empty state, 05의 interaction 규칙에 반영한다.

> **핵심 인사이트**: 필터는 장식용 control이 아니라, 결과가 어떤 근거로 바뀌는지 설명하는 작은 데이터 계약이다.

### QA-08. 보고서의 기간·시간대·export 결과는 누가 재현해도 같은 의미를 갖는가?

**왜 지금 묻는가**
공유되는 수치는 화면에서 멋지게 보이는 것보다, 같은 조건에서 다시 만들 수 있는지가 중요하다. 특히 export를 toast로만 끝내면 완료 여부와 공유 범위를 확인할 수 없다.

**FACT — AIoT 예시에서 확인한 것**
Reports는 기간·site 필터, KPI, 추이, 표, export 진입점을 보인다. export는 toast만 표시하며 실제 파일 다운로드는 확인되지 않는다.

**ASSUMPTION — 교육용 합의안**
리포트에는 applied scope, 기준 시간대, 데이터 freshness, 지표의 간단한 정의를 함께 보인다. export는 `Preparing → Ready / Failed` 상태를 별도 표시하고, 실습에서는 `Simulated export`라고 명시한다.

**OPEN — 실제 사례에서 확인할 것**
파일 형식, 접근 권한, 다운로드 URL의 만료, 보존·공유 정책, 집계의 기준 시각과 timezone은 데이터·보안·운영 담당이 정한다.

**다음 반영**
02의 report 요구사항, 03의 report data/state, 06의 export job과 수신 기준에 반영한다.

> **핵심 인사이트**: 공유용 수치의 신뢰는 큰 숫자가 아니라, 범위·시간·생성 결과를 재현할 수 있는지에서 시작된다.

### QA-09. Settings에서 수정 가능한 것과 ‘위험 변경’의 경계는 무엇인가?

**왜 지금 묻는가**
관리자 화면이라고 해서 모든 항목을 즉시 변경 가능하게 만들면, 운영 정책·권한·복구 책임이 숨겨진다.

**FACT — AIoT 예시에서 확인한 것**
Settings에는 역할, 알림 규칙, 채널, 사이트 설정, Danger Zone이 있다. 각 항목이 조회만 필요한지 CRUD까지 필요한지, 영향과 rollback을 어떻게 다룰지는 확정되지 않았다.

**ASSUMPTION — 교육용 합의안**
각 설정에는 `조회 / 편집 / 승인 필요 / 되돌리기 가능` 여부를 구분한다. 삭제·임계치 변경처럼 영향이 큰 행동은 현재값·변경값·영향 대상·확인 행동을 먼저 보여 주고, Cancel은 아무 변경도 만들지 않게 한다.

**OPEN — 실제 사례에서 확인할 것**
관리자 역할, 승인 단계, 정책 적용 시점, 감사 로그, rollback 가능 여부와 복구 절차는 실제 운영 정책으로 확정해야 한다.

**다음 반영**
02의 설정 범위, 03의 danger action, 06의 변경 승인·audit 기준에 반영한다.

> **핵심 인사이트**: 관리자 기능의 성숙도는 편집 버튼 수가 아니라, 변화의 영향과 복구 가능성을 얼마나 명확히 다루는가에 있다.

### QA-10. 화면에 필요한 데이터와 실제 API·운영 데이터 계약은 누가 확정하는가?

**왜 지금 묻는가**
UI에 표시된 필드 목록을 바로 API 명세나 데이터베이스 스키마로 넘기면, prototype의 편의상 값이 실제 업무 규칙으로 고정될 수 있다.

**FACT — AIoT 예시에서 확인한 것**
화면의 Device, Alert, Command, Campaign, Report, Settings 필드는 UI 표시 후보이며, API endpoint·DB schema·telemetry protocol은 확정되지 않았다.

**ASSUMPTION — 교육용 합의안**
화면별로 “필수 표시 필드 / 갱신 시점 / 값이 없을 때의 표현 / 데이터 출처 책임자”를 적는다. 현재 수집할 수 없는 값은 정상값으로 가정하지 않고 `Unknown` 또는 `Pending`으로 유지한다.

**OPEN — 실제 사례에서 확인할 것**
데이터 오너, source of truth, 수집 주기, 보존 기간, 개인정보·보안 분류, API 계약과 오류 응답은 개발·플랫폼·데이터 책임자가 정한다.

**다음 반영**
01의 evidence/assumption, 03의 content model, 06의 data·API handoff에 반영한다.

> **핵심 인사이트**: 좋은 handoff는 데이터 빈칸을 감추지 않고, 누가 그 빈칸을 채울지까지 전달한다.

### QA-11. 작업의 완료는 화면 변화인가, 다시 확인 가능한 기록까지인가?

**왜 지금 묻는가**
toast와 성공 애니메이션은 시연에는 충분할 수 있지만, 실제 운영에서는 새로고침 후에도 누가 무엇을 했는지 확인할 수 있어야 한다.

**FACT — AIoT 예시에서 확인한 것**
명령·캠페인·설정은 브라우저 메모리 시연이며 새로고침 후 사라진다. export도 toast만 보인다. 위험 행동에는 audit와 결과 이력이 필요하다는 보정 방향이 도출됐다.

**ASSUMPTION — 교육용 합의안**
명령·배포·설정·export는 toast만으로 완료로 보지 않는다. 화면 안에 결과 상태, 대상, 시각, 다음 행동 또는 이력 위치를 제공한다. 실제 저장이 없는 데모라면 그 한계를 명시한다.

**OPEN — 실제 사례에서 확인할 것**
감사 데이터의 저장 위치·보존 기간, 권한별 조회 범위, 실패 재처리, 결과 정정·삭제 정책은 보안·운영·개발이 함께 결정한다.

**다음 반영**
05의 state 표현, 06의 audit/acceptance mapping, 07의 배포 검증 체크에 반영한다.

> **핵심 인사이트**: 시연의 ‘성공’과 운영의 ‘완료’는 다르다. 운영의 완료에는 재확인 가능한 결과와 책임의 흔적이 필요하다.

## Q&A를 실제 업무 기록으로 만드는 최소 규칙

### 질문을 네 가지로 기록한다

| 기록 항목 | 남길 내용 | 피해야 할 기록 |
| --- | --- | --- |
| 근거 | 화면/문서 ID와 현재 확인 사실 | “그냥 그렇게 보인다” |
| 질문 | 어떤 결정이 필요한지와 결정하지 않을 때의 영향 | “이거 어떻게 할까요?”만 적기 |
| 답변 | 교육용 가정 또는 실제 책임자의 결정 | 추정을 확정 사실처럼 쓰기 |
| 후속 | 갱신할 문서, 검증 시나리오, 담당·기한 | 답변만 적고 산출물에 반영하지 않기 |

### 회의 중에는 이 순서로 묻는다

1. **범위** — 이번에 실제로 검증할 행동과 시연에만 남길 행동은 무엇인가?
2. **운영 규칙** — 누가, 어떤 조건에서, 어떤 결과까지 책임지는가?
3. **예외** — 실패·0건·권한 부족·데이터 지연이면 사용자는 무엇을 보며 무엇을 하는가?
4. **증거** — 성공/실패를 무엇으로 판단하고 어디에 기록하는가?
5. **환류** — 답변은 어느 산출물과 검수 시나리오를 바꾸는가?

### 기록 상태는 단순하게 유지한다

| 상태 | 의미 | 다음 행동 |
| --- | --- | --- |
| `FACT` | 출처와 책임자가 있는 사실 | 근거 ID·날짜를 남긴다. |
| `ASSUMPTION` | prototype 일관성을 위한 임시 가정 | 실제 정책으로 단정하지 않고 화면에 경계를 표시한다. |
| `DECISION` | 책임자가 승인한 선택 | 영향 문서와 acceptance criteria를 갱신한다. |
| `OPEN` | 실제 제품화 전에 결정이 필요한 항목 | 담당자·결정 기한·영향 문서를 지정한다. |

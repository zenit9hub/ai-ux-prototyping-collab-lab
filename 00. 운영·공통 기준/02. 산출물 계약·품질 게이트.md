# 산출물 계약·품질 게이트

## 1. 공통 산출물 계약

모든 canonical 문서는 다음 frontmatter와 본문 구조를 사용한다.

```yaml
case_id: example-case
stage: 01-research
status: draft # draft | review | accepted | superseded
owner: Research / Evidence Owner
reviewer: Product Decision Owner
upstream_baseline: ../00. 운영·공통 기준/...
decision_ids: [DEC-001]
next_consumer: Product Decision Owner
ai_contribution: 도구·프롬프트·사람 검토 범위
```

본문에는 최소한 아래 항목이 있어야 한다.

1. 이번 단계의 목적
2. 입력 baseline과 사용 범위
3. FACT / ASSUMPTION / DECISION / OPEN
4. 이번 단계의 핵심 산출물
5. 품질 게이트 self-check
6. 다음 단계 전달 패키지

## 2. 단계별 계약

| 단계 | 최소 산출물 | 다음 단계 전달물 | 품질 게이트 |
| --- | --- | --- | --- |
| 01 Research | evidence map, 요구사항 후보, 가설·리스크·질문 | 근거 ID, 문제 정의, 검증 필요 항목 | 출처·날짜·사실/가설 구분, 핵심 문제 우선순위 |
| 02 PRD | 사용자, 목표, 범위/비범위, 요구사항, acceptance criteria | 요구사항 ID, priority, non-goal, acceptance criteria | 각 요구사항에 왜/누구/완료 기준 존재 |
| 03 IA | menu, content model, task flow, state/exception | screen ID, flow ID, state rules, data display fields | PRD 요구사항 누락 없음, 정상/빈/오류/권한 상태 포함 |
| 04 Wireframe | 화면 구조, 정보 위계, CTA, annotation | screen layout, component requirement, action placement | 핵심 과업을 화면으로 수행 가능, 사용자의 다음 행동 명확 |
| 05 Hi-Fi | canonical design, token, component, state, asset manifest | Figma/Claude link, frame/page ID, frozen version, UI states | wireframe 의도 보존, component 일관성, 접근성·반응형 확인 |
| 06 Handoff | handoff index, state/responsive/asset/open question, receiver acknowledgement | 구현 기준 링크와 acceptance mapping | 수신자가 구현 시작 가능, blocker와 결정 필요 항목 기록 |
| 07 Deploy & Verify | 정적 site source, Pages workflow, deployed URL, 검토 이슈 | 최신 검증 URL과 피드백 추적 | artifact 최상단 index, 상대 경로, mock data·민감정보 구분, 배포 성공 |

## 3. 게이트 운영 방식

| 결과 | 의미 | 다음 행동 |
| --- | --- | --- |
| Pass | 다음 단계가 이 자료만으로 시작 가능 | `accepted`로 변경, downstream handoff |
| Pass with assumptions | 명시된 가정을 전제로 시작 가능 | ASSUMPTION/OPEN을 문서와 PR에 유지 |
| Rework | 근거·범위·상태·전달물이 부족 | owner가 수정 후 재검토 |
| Blocked | 외부 결정·권한·자료가 없어 진행 불가 | `OPEN`과 blocker owner를 기록 |

## 4. 공통 품질 루브릭

| 관점 | 질문 |
| --- | --- |
| Traceability | 직전 입력, 결정 ID, 다음 전달물이 연결되는가? |
| Factuality | FACT와 ASSUMPTION, 예시 데이터와 실제 데이터가 분리되는가? |
| Completeness | 정상·빈·오류·권한·실패 상태가 필요한 단계에서 빠지지 않았는가? |
| Usability | 다음 역할이 별도 해석 없이 시작할 수 있는가? |
| Consistency | 용어, ID, 상태명, 범위가 이전 단계와 충돌하지 않는가? |
| Safety | 위험 행동, 개인정보, 비밀값, 접근성 제약이 반영되는가? |

## 5. Hi-Fi와 handoff의 특별 규칙

### 05. Hi-Fi Design

- Figma 또는 Claude Design 중 **한 개의 canonical source**를 사례마다 지정한다.
- 다른 도구 결과는 비교·탐색 산출물이며 canonical로 오인되지 않게 상태를 표시한다.
- canonical manifest에는 URL, page/frame ID, 동결일, version, owner, 접근 권한, 연관 PR을 기록한다.

### 06. 개발 전달·퍼블리싱

- Hi-Fi 파일을 다시 복사하거나 export만 쌓지 않는다. 다만 구현 수신자가 동작·시각을 확인해야 하는 경우, 동결 버전당 하나의 standalone HTML을 `reference-only`로 handoff package에 포함할 수 있다.
- handoff manifest에서 05의 canonical source를 직접 가리킨다.
- standalone HTML이 있으면 생성 도구, 동결일, 해시, `reference-only` 용도, 기준 manifest와의 연결을 기록한다.
- 화면별 상태, 반응형, asset, 텍스트, 미해결 질문, acceptance mapping을 보완한다.
- 퍼블리싱/프론트엔드 수신자는 “수신 가능” 또는 “blocker”를 명확히 기록한다.

### 07. 배포·검증 (GitHub Pages)

- 정적 site는 index.html, CSS, JavaScript, mock data를 분리하고 GitHub Pages artifact 최상단에 index.html을 둔다.
- 배포 source는 생성 도구의 대형 bundle 대신 사람이 유지보수 가능한 구조를 사용한다.
- Pages URL은 검토용 공유 결과이며, 실제 운영 데이터·비밀값·실장비 제어를 포함하지 않는다.
- 배포 화면에서 발견한 차이는 Issue 또는 CR로 기록해 가장 이른 영향 단계부터 재검토한다.

## 6. PR 품질 게이트 체크

- [ ] upstream baseline이 `accepted` 상태이며 경로/commit/PR reference가 있다.
- [ ] 이번 단계의 최소 산출물과 metadata가 있다.
- [ ] FACT / ASSUMPTION / DECISION / OPEN이 구분된다.
- [ ] next consumer와 전달 패키지가 적혀 있다.
- [ ] 영향을 받는 하위 단계가 있으면 기록했다.
- [ ] AI가 기여한 범위와 사람 검토 범위를 적었다.

Harness engineering

AI에게 자유를 주는 대신, 작업 환경을 고정합니다

하네스는 프롬프트 문장 하나가 아닙니다. AI와 사람이 같은 문제를 풀고, 결과를 검토 가능한 형태로 남기도록 입력·제약·출력·평가를 정렬하는 운영 방식입니다.

  • 입력직전 accepted baseline과 그 근거를 명시
  • 기록FACT · ASSUMPTION · DECISION · OPEN을 분리
  • 검토다음 단계 수신자가 “이것으로 시작 가능한가”를 확인
  • 변경상위 변경은 CR로 영향 범위를 기록하고 재검토
CONTEXT 무엇을 알고 있는가 근거, baseline, 사용자, 업무 맥락
INTENT 무엇을 결정할 것인가 목적, 문제, 범위, 다음 소비자
CONSTRAINT 무엇을 넘지 않는가 보안, 접근성, 정책, 비범위
OUTPUT 어떤 형식으로 남길 것인가 문서, manifest, 표, 링크, 상태
EVALUATION 무엇으로 통과를 판단할까 Quality Gate, reviewer, acceptance

Hi-Fi는 한 번만 만들고, 전달과 배포 검증은 얇게 연결합니다

06은 디자인의 복제 보관함이 아닙니다. 기준 원본은 05에 유지하고, 07은 표준 정적 source를 GitHub Pages에 배포해 버전·상태·에셋·반응형·미해결 질문을 실제 화면에서 확인합니다.

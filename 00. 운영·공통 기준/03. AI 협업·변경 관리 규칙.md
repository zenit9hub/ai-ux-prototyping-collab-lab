# AI 협업·변경 관리 규칙

## 1. AI 작업 입력 패턴

AI 작업 요청은 아래 다섯 요소를 기본으로 한다.

```text
Context: 승인된 upstream baseline과 사례 맥락
Intent: 이번 단계에서 답해야 할 질문
Constraint: 범위·사실성·보안·도구·시각 제약
Output Format: 요구 파일·표·ID·링크 형식
Evaluation Criteria: 다음 단계가 수신 가능한지 판단할 기준
```

프롬프트를 길게 쓰는 것보다, 승인된 입력과 금지된 가정을 명확히 하는 것을 우선한다.

## 2. AI 결과의 세 가지 상태

| 상태 | 의미 | 저장 위치 |
| --- | --- | --- |
| exploration | 비교·실험·대안 생성 | 작업 브랜치 또는 명시적 실험 파일 |
| candidate | 사람이 검토할 후보 | 현재 단계 폴더, `draft/review` |
| canonical | 사람이 승인한 기준 | 현재 단계 폴더, `accepted` |

AI가 만든 결과라고 해서 exploration에 머물 필요는 없지만, 사람의 책임자와 quality gate 없이 canonical이 될 수 없다.

## 3. AI 작업 기록

다음 중 하나를 문서 frontmatter 또는 PR에 남긴다.

```yaml
ai_contribution:
  tool: Figma Make | Claude Design | Codex | 기타
  prompt_or_run: 링크 또는 식별자
  used_inputs: [accepted baseline 경로]
  human_review: 검토한 역할과 수정 범위
```

실제 고객 데이터, 접근 토큰, 내부 비밀, 인증 정보는 AI 입력·저장소·스크린샷에 넣지 않는다.

## 4. 변경 요청(CR) 규칙

변경 요청은 단순 오타 수정이 아니라 다음에 해당할 때 만든다.

- 요구사항, 우선순위, 비범위, 수용 기준의 변경
- IA, state, 사용자 흐름, component contract의 변경
- canonical Hi-Fi, 반응형, asset, handoff 기준의 변경
- 다음 단계 산출물의 해석을 바꿀 수 있는 결정 변경

### CR 최소 형식

```markdown
# CR-### 변경 요청

## 변경 이유

## 변경 전 baseline

## 변경 후 제안

## 영향 단계와 산출물

## 필요한 reviewer / 결정자

## 재검토 완료 조건
```

## 5. 하위 영향 규칙

| 바뀐 단계 | 우선 검토할 하위 단계 |
| --- | --- |
| Research | PRD, IA, Wireframe, Hi-Fi, Handoff |
| PRD | IA, Wireframe, Hi-Fi, Handoff |
| IA | Wireframe, Hi-Fi, Handoff |
| Wireframe | Hi-Fi, Handoff |
| Hi-Fi | Handoff |
| Handoff | 구현팀 이슈 및 수신 확인 |

상위 기준이 변경되면 하위 산출물은 즉시 삭제하지 않는다. `superseded` 또는 `review required` 상태로 남겨 영향과 이력을 추적한다.

## 6. GitHub 운용 최소안

- `main`: accepted baseline만 유지
- 작업 브랜치: `work/<case-id>/<NN>-<stage-name>`
- PR: 한 사례의 한 단계 또는 하나의 CR에 집중
- Issue label: `stage:01~06`, `type:decision`, `type:change`, `type:handoff`, `risk`
- CODEOWNERS: GitHub 사용자/팀이 정해진 뒤 활성화
- Actions: 구조·링크·metadata만 자동 검증, 품질 판정은 사람 reviewer가 수행

## 7. 디자인 도구 연결

Figma와 Claude Design의 결과는 URL 하나만 적고 끝내지 않는다. 아래 정보를 manifest로 남긴다.

- canonical 여부
- tool/workspace/file URL
- page 또는 frame ID
- 동결일과 version
- source of truth owner
- 검토한 state와 breakpoint
- 내보내야 할 asset 및 라이선스/사용 제약
- 관련 PR·Issue·handoff 문서

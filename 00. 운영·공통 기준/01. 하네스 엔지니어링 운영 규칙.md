# 하네스 엔지니어링 운영 규칙

## 1. 하네스의 정의

이 저장소에서 하네스 엔지니어링은 “좋은 프롬프트를 많이 모으는 일”이 아니다. 사람과 AI가 같은 기준으로 반복 작업할 수 있게 **입력·제약·출력·검증·인수인계 환경을 설계하는 일**이다.

```text
accepted upstream baseline
  + Context
  + Intent
  + Constraint
  + Output Format
  + Evaluation Criteria
→ 사람·AI 공동 작업
→ reviewer 검토
→ accepted baseline
→ 다음 단계의 유일한 입력
```

## 2. 하네스의 네 층

| 층 | 질문 | 저장소에서의 구현 |
| --- | --- | --- |
| Context | 무엇을 만들며 어떤 근거를 쓰는가? | R&R, 사례 브리프, accepted upstream link |
| Constraint | 무엇을 하지 말아야 하는가? | 범위/비범위, 사실성, 보안, 디자인·기술 guardrail |
| Interface | 무엇을 어떤 형태로 넘기는가? | stage README, template, metadata, canonical manifest |
| Evaluation | 다음 사람이 이 자료로 시작 가능한가? | quality gate, PR checklist, downstream review |

## 3. 모든 단계에 고정하는 여섯 항목

각 단계 README와 canonical 문서는 다음 항목을 빠뜨리지 않는다.

1. **입력 기준**: 어느 문서·commit·PR이 baseline인가
2. **목적**: 이번 단계가 해결할 질문은 무엇인가
3. **제약**: 사실성, 범위, 보안, 도구, 디자인 관련 제한은 무엇인가
4. **요구 출력**: 다음 단계에 넘길 파일·링크·ID·형식은 무엇인가
5. **품질 게이트**: 통과·보완·반려를 어떻게 판단하는가
6. **다음 인수인계**: 누가 무엇을 수신하고 어떤 질문을 할 수 있는가

## 4. 정보 상태 언어

| 표기 | 의미 | 다음 단계 입력 가능 여부 |
| --- | --- | --- |
| FACT | 출처·날짜·근거가 확인된 사실 | 가능 |
| ASSUMPTION | 설계를 진행하기 위한 가정 | 조건부, 가정으로 표시할 때만 가능 |
| DECISION | 책임자가 승인한 선택 | 가능 |
| OPEN | 아직 결정·검증이 필요한 항목 | 해결 전에는 blocker 또는 명시적 제한 |
| draft | 작성 중 산출물 | 불가 |
| review | 검토 중 산출물 | 불가 |
| accepted | 다음 단계가 사용할 기준 | 가능 |
| superseded | 대체된 이전 기준 | 불가, 이력 참고만 가능 |

## 5. AI 에이전트의 권한 경계

AI는 아래 행동을 할 수 있다.

- 입력 자료 구조화, 요약, 누락 항목 식별
- 정해진 output format의 초안·변형안 생성
- 요구사항·화면·상태·handoff 간 traceability 점검
- quality gate의 기계적 항목을 self-check

AI는 아래 행동을 할 수 없다.

- 실제 리서치가 없는데 사실·고객 발언·운영 데이터를 만들어 내기
- 사람 승인 없이 `accepted` 상태로 바꾸기
- 범위 밖 폴더·upstream baseline을 무단 수정하기
- 보안·권한·개인정보·법무 요구를 임의로 확정하기

## 6. 최소 자동화와 사람의 판단

자동 검증은 다음처럼 객관적인 구조에 한정한다.

- 필수 폴더와 README 존재
- canonical 문서의 metadata·상태값·입력 reference 존재
- 상대 링크와 파일 경로 규칙
- PR의 단계·품질 게이트 항목

사람이 판단해야 할 항목은 다음과 같다.

- 문제 정의의 타당성
- 근거의 충분성
- UX 품질과 정보 위계
- AI 결과 중 무엇을 채택할지
- 실제 운영·개발·보안 제약의 적합성

## 7. 변경 시 행동 규칙

상위 단계의 입력이 달라지면, 하위 단계의 결과가 자동으로 유효하다고 가정하지 않는다. `CR-###`으로 변경을 기록하고 아래를 적는다.

- 바뀐 baseline과 변경 이유
- 영향을 받는 `case_id`와 stage
- 다시 검토할 canonical 산출물
- 임시로 허용하는 가정 또는 blocker
- 재수신 reviewer와 목표 완료일

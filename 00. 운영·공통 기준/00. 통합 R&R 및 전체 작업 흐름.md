# 통합 R&R 및 전체 작업 흐름

## 1. 이 문서의 목적

이 문서는 공동 실습 저장소의 **유일한 운영 기준**이다. 폴더마다 누가 무엇을 결정하고, 어떤 입력을 받아 무엇을 다음 단계에 넘기는지 설명한다.

강의 Vault는 이론·예시·진행을 위한 학습 자료이고, 이 GitHub 저장소는 AI 에이전트와 사람이 실제 업무에 가깝게 협업하는 작업 시스템이다. 두 공간의 역할을 섞지 않는다.

| 구분 | 강의 Vault | 이 GitHub 저장소 |
| --- | --- | --- |
| 주 역할 | 이론, 사례, 강의 진행, 참고 자료 | 실제 실습 작업, 검토, 인수인계, 변경 이력 |
| 기준 | 강사가 제공하는 교육 맥락 | accepted baseline과 Pull Request 기록 |
| 결과물 | 학습 이해를 돕는 자료 | 현업으로 가져갈 수 있는 작업·협업 템플릿 |
| 의존성 | 수업 중 주 참고 | 수업 후에도 독립적으로 작동 |

## 2. 전체 흐름

```text
00. 운영·공통 기준
      ↓ 공통 Context / Guardrail / Quality Gate
01. 요구사항 수집·리서치
      ↓ 증거·가설·문제 정의
02. PRD
      ↓ 목표·범위·요구사항·수용 기준
03. IA
      ↓ 메뉴·콘텐츠·Task Flow·State/Exception
04. Wireframe
      ↓ 정보 위계·레이아웃·행동 annotation
05. Hi-Fi Design (Figma Make, Claude Design)
      ↓ canonical design·token·component·state
06. 개발 전달·퍼블리싱
      ↓ handoff manifest·구현 질의·수신 확인
07. 배포·검증 (GitHub Pages)
      ↓ 정적 검증 source·배포 URL·검토 피드백
```

각 화살표는 “문서를 옮긴다”가 아니라 **승인된 입력을 다음 역할이 시작 가능한 형태로 인수인계한다**는 뜻이다. 다음 단계는 직전 단계의 `accepted` 산출물만 baseline으로 사용할 수 있다.

## 3. 폴더의 의미

폴더는 직무 권한 장벽이 아니라 **산출물의 성숙 단계**다. 교육에서는 한 명의 수강생이 모든 단계를 순차 수행할 수 있고, 현업에서는 여러 사람이 역할을 나누어 수행할 수 있다.

| 폴더 | 책임 범위 | 포함 | 포함하지 않음 | 다음 연결 |
| --- | --- | --- | --- | --- |
| `00. 운영·공통 기준` | control plane | R&R, 하네스, 템플릿, 품질 게이트, AI·변경 규칙 | 사례별 실제 산출물 | 모든 단계의 입력 계약 |
| `01. 요구사항 수집·리서치` | 근거와 문제 정의 | 자료 출처, 관찰, 가설, 질문, 요구사항 후보 | 확정된 제품 정책이나 디자인 | 02 PRD |
| `02. PRD` | 제품 의사결정 | 사용자·목표·범위·비범위·우선순위·수용 기준 | 화면 구조나 Hi-Fi 확정 | 03 IA |
| `03. IA` | 구조와 과업 흐름 | 메뉴, 콘텐츠 모델, user flow, state, exception | 화면의 시각 완성도 | 04 Wireframe |
| `04. Wireframe` | 저충실도 화면 구조 | 정보 우선순위, CTA, 레이아웃, 상태별 구조 | 최종 색·타이포·asset | 05 Hi-Fi |
| `05. Hi-Fi Design (Figma Make, Claude Design)` | 시각·component 기준 | canonical Figma/Claude 결과, token, component, 상태, asset manifest | 구현 코드의 복사본 | 06 Handoff |
| `06. 개발 전달·퍼블리싱` | 구현 수신과 해석 확인 | 기준 링크, 화면·상태·반응형·asset·질의·수신 확인 | 별도 개발 source code | 07 배포·검증 및 별도 개발 저장소 또는 팀 |
| 07. 배포·검증 (GitHub Pages) | 배포된 결과 검증 | 정적 site source, GitHub Pages workflow, 배포 URL, 검토 피드백 | 디자인 원본·운영 서비스의 비밀값 | 검토자와 별도 개발 저장소 또는 팀 |

## 4. 책임 기능(R&R)

역할은 특정 개인의 직함이 아니라 책임 기능이다. 한 사람이 여러 역할을 맡을 수 있지만, 무엇을 근거로 어떤 결정을 했는지는 역할별로 구분해 기록한다.

| 책임 기능 | 실습에서의 역할 | 현업 전환 시 역할 | 주요 책임 | 승인하지 않는 것 |
| --- | --- | --- | --- | --- |
| Repository Steward | 강사·운영자 | 디자인 운영 리드 또는 repo 관리자 | `00` 기준, 템플릿, 구조, 예외 운영 | 사업 우선순위·실제 고객 사실 |
| Facilitator | 강사 | 팀 리드·코치 | 질문, 비교, 피드백, 품질 게이트 안내 | 참여자 대신 최종 산출물 작성·승인 |
| Research / Evidence Owner | 수강생 역할 | 리서처·PO | 근거 수집, 출처·날짜, 사실/가설 분리 | PRD의 최종 범위 결정 |
| Product Decision Owner | 수강생 역할 | PO·PM·의사결정자 | 문제, 대상, 우선순위, 범위, 수용 기준 | IA·UI의 세부 표현 단독 결정 |
| UX / IA Owner | 수강생 역할 | UX Architect·서비스 기획 | 정보 구조, task flow, state, exception | 브랜드·시각 시스템의 최종 승인 |
| UX / UI Owner | 수강생 역할 | UX/UI Designer | wireframe, visual hierarchy, interaction annotation | 실제 개발 권한·운영 정책 승인 |
| AI Design Operator | 수강생 역할 | UI Designer·AI 협업자 | AI 입력 구성, 변형안 생성, 결과 비교, canonical 후보 제안 | AI 결과의 무검토 승인 |
| Publishing / Frontend Receiver | 동료 reviewer | 퍼블리셔·프론트엔드 담당 | 구현성 검토, 모호성 질의, handoff 수신 확인 | 제품 범위·디자인 기준의 일방 변경 |
| Peer / Downstream Reviewer | 다음 단계 담당 수강생 | 후속 역할 담당자 | “이 입력으로 시작 가능한가”를 검토 | 본인 소유가 아닌 결정의 독단 승인 |
| AI Agent | 도구 | 도구 | 제약 안의 초안·구조화·검증 보조 | 사실·정책·수용 기준의 승인 |

### 핵심 원칙

1. 강사는 **운영 기준의 steward**이지, 모든 실무 결정을 대신하는 승인자가 아니다.
2. 다음 단계 수신자가 직전 산출물의 reviewer가 된다. 이렇게 해야 인수인계 품질이 실제로 검증된다.
3. AI 에이전트의 결과는 `draft`일 뿐이며, 사람의 검토·결정 없이 `accepted`가 될 수 없다.
4. 폴더 접근은 협업을 막는 권한 장벽이 아니라 책임과 결과물의 경계를 드러낸다.

## 6. canonical 기준과 변경 관리

### canonical 산출물

- 한 사례·한 단계에는 하나의 canonical 산출물을 둔다.
- Git commit과 Pull Request가 버전 이력이다. `_final_v3`, `최종_진짜` 같은 파일 복제 방식은 사용하지 않는다.
- Figma/Claude Design은 `05`의 canonical manifest에 URL, 페이지/프레임 ID, 동결일, 버전을 기록한다.
- `06`은 `05`의 복사본을 저장하지 않고 manifest로 참조한다.
- 07은 06의 구현 전달 기준을 반영한 정적 검증 source만 배포한다. 제품·디자인 기준을 바꾸려면 CR로 상위 단계에 되돌린다.

### 변경 요청

상위 결정이 바뀌면 조용히 하위 문서만 수정하지 않는다.

1. `CR-###` 변경 요청을 만들고 변경 이유와 영향 범위를 적는다.
2. 가장 이른 영향을 받는 단계의 owner가 baseline을 갱신한다.
3. 영향받는 하위 단계는 `review`로 되돌리고 재검토 대상임을 표시한다.
4. 새 기준이 `accepted`된 뒤에만 downstream 작업을 재개한다.

## 7. 수업에서 현업으로의 전환

| 구분 | 공동 실습 | 현업 도입 |
| --- | --- | --- |
| 사례 | 강사가 제공한 안전한 시나리오와 예시 자료 | 실제 업무 문제·조직 제약·고객 근거 |
| 역할 | 한 사람이 여러 책임 기능을 순차 수행 가능 | PO/리서치/UX/UI/퍼블리싱이 역할 분담 |
| 리뷰 | 강사·동료 중심 | 다음 단계 수신자·승인권자 중심 |
| AI 사용 | 도구 비교와 입력 설계 학습 | 보안·개인정보·브랜드·법무 제약 반영 |
| GitHub | branch/PR/handoff 습관 학습 | CODEOWNERS, Issue, release, 개발 저장소 연계 |

수강 후에도 이 저장소를 clone하여 실제 사례의 `case_id`만 바꾸고, 공통 하네스와 단계별 계약을 재사용할 수 있다.

## 8. 이 문서의 변경 권한

이 문서는 `00`의 accepted 기준이다. 구조·역할·품질 게이트를 바꾸는 변경은 Repository Steward와 관련 역할의 검토를 거쳐야 한다. 사례별 산출물만 변경하는 PR에서 이 문서를 임의로 수정하지 않는다.

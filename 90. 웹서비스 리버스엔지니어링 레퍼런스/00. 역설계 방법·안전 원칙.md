# 웹서비스 역설계 방법·안전 원칙

## 1. 목적과 경계

웹서비스 역설계는 화면을 그대로 재현하는 작업이 아니라, 공개적으로 관찰 가능한 UX·정보 구조·업무 흐름을 **근거 기반의 가설**로 정리하는 작업이다. 결과물은 “그 서비스가 실제로 이렇게 운영된다”는 명세가 아니라, “우리 사례에서 이런 문제와 패턴을 검토해 볼 수 있다”는 학습 입력이다.

## 2. 수집 방법은 자동화가 아니라 안전성부터 결정한다

기본값은 **브라우저 기반 관찰**이다. 화면·메뉴·사용자 과업·상태·예외를 제한된 시나리오 안에서 기록하는 것만으로도 UX 역설계에는 충분한 경우가 많다.

자동 수집은 아래 조건을 모두 충족할 때만 보조적으로 사용한다.

| 판단 항목 | 허용 기준 | 중단 또는 전환 기준 |
| --- | --- | --- |
| 접근 범위 | 로그인 없이 접근 가능한 공개 페이지 | 로그인, 결제, 개인화, 관리자·계정 페이지가 필요함 |
| 이용 조건 | 이용약관과 `robots.txt`를 확인했고 자동 접근을 금지하지 않음 | 수집 금지·불명확, 추가 허가 필요, rate limit 경고 |
| 수집 목적 | 메뉴·제목·URL 구조·공개 정보 구조의 요약 | 원문 대량 복제, 데이터베이스 구축, 경쟁 서비스 복제 |
| 요청 방식 | 단일 도메인, 저빈도, 얕은 깊이, 정해진 페이지 수 | 403/429, CAPTCHA, 세션·토큰 요구, 동적 API 탐색 필요 |
| 결과 보관 | URL·시각·요약·관찰 ID만 보관 | 이미지·코드·대량 원문·개인정보를 저장해야 함 |

> [!important]
> `robots.txt`는 자동 접근 범위에 대한 신호일 뿐, 이용약관·저작권·계약상 허용 여부를 대신 판단하지 않는다. 하나라도 불명확하면 자동 수집을 하지 않고, 사용자에게 공개 화면 관찰 자료 또는 별도 권한을 요청한다.

## 3. 권장 수집 프로토콜

### A. 사전 확인

1. 대상 URL, 서비스명, 관찰 목적, 수강생이 배우려는 업무 패턴을 적는다.
2. 공개 접근 여부와 약관·`robots.txt`를 확인한다.
3. 수집 방식을 `브라우저 관찰`, `제한적 정적 수집`, `진행 불가` 중 하나로 결정한다.
4. 로그인, 결제, 개인정보, 계정 설정, 고객 데이터, API 응답은 범위에서 명시적으로 제외한다.

### B. 관찰 또는 제한적 자동 수집

**브라우저 관찰**에서는 대표 과업 3~5개만 선택한다. 예를 들어 탐색, 검색/필터, 상세 확인, 생성/변경 전 확인, 오류/빈 상태의 순서로 관찰한다.

**제한적 정적 수집**을 허용한 경우에도 다음 기본값을 넘기지 않는다.

- 같은 도메인의 공개 HTML 페이지만 대상으로 한다.
- 최대 깊이 2, 최대 30개 페이지, 요청 간 최소 2초 간격을 기본값으로 둔다.
- `403`, `429`, CAPTCHA, 로그인 유도, 세션·토큰 요구가 나타나면 즉시 중단한다.
- form 제출, 검색어 대량 변경, 장바구니·결제, 계정 생성, API endpoint 추측은 수행하지 않는다.
- 원문·이미지·소스코드를 저장하지 않고, 정보 구조와 관찰 요약만 근거 ID와 함께 남긴다.

### C. 사실·해석·결정의 분리

| 표기 | 기록 기준 | 예시 |
| --- | --- | --- |
| `FACT` | 화면 또는 공개 문서에서 직접 확인했고 URL·시각이 있음 | “상단 메뉴에 요금제와 도움말 진입점이 보임” |
| `ASSUMPTION` | 관찰한 사실을 바탕으로 한 UX·업무 해석 | “셀프서비스 문의를 줄이려는 의도로 보임” |
| `DECISION` | 수강생의 사례에 적용하기로 사람 책임자가 선택함 | “우리 서비스는 검색 결과에 비교 기능을 포함하기로 함” |
| `OPEN` | 화면만으로 확인할 수 없어 추가 판단이 필요한 항목 | “추천 정렬의 실제 데이터 기준은 확인 불가” |

외부 서비스의 화면을 근거로 `DECISION`을 자동으로 만들지 않는다. `DECISION`은 항상 자신의 제품 책임자 또는 실습 reviewer가 승인한 선택이어야 한다.

## 4. 사례별 산출물의 작성 순서

| 순서 | 파일 | 핵심 질문 | 다음 입력 |
| --- | --- | --- | --- |
| 00 | 범위·출처·권리 확인 | 무엇을, 어떤 권한으로, 어떤 방식으로 관찰하는가? | 관찰 범위 |
| 01 | 관찰 근거 맵 | 실제로 무엇을 보았고 출처는 어디인가? | FACT / ASSUMPTION / OPEN |
| 02 | 역설계 PRD | 이 서비스가 해결하려는 문제와 핵심 과업은 무엇으로 보이는가? | 요구사항 가설 |
| 03 | IA·User Flow | 정보·화면·상태·예외는 어떤 관계인가? | screen / flow / state 가설 |
| 04 | Wireframe·UX 패턴 | 사용자의 판단·행동을 돕는 정보 위계와 control은 무엇인가? | 재사용 가능한 UX 패턴 |
| 05 | 시니어 Q&A·결정 기록 | 화면만으로 알 수 없는 운영·권한·데이터·실패 조건은 무엇인가? | OPEN과 결정 요청 |
| 06 | 현업 적용 포인트 | 무엇을 참조하고, 무엇을 우리 문제에 맞게 바꿀 것인가? | 새 `01~07` 사례의 입력 후보 |

## 5. 금지 및 주의 범위

- 계정 생성·로그인·유료 결제·개인화 데이터·관리자 기능을 관찰하거나 우회하지 않는다.
- 공개 화면에 우연히 노출된 개인정보, 고객 정보, 토큰, 내부 URL을 복사·기록·공유하지 않는다.
- 프런트엔드 소스, JavaScript bundle, 숨겨진 API, 비공개 endpoint를 분석 대상으로 삼지 않는다.
- 서비스 로고, 이미지, 긴 원문 카피, 디자인 asset을 자신의 결과물에 재사용하지 않는다.
- 사실과 가설을 구분하지 않은 채 “경쟁사가 이렇게 하니 우리도 해야 한다”는 결론을 내리지 않는다.

## 6. 역설계 결과를 현업으로 옮길 때의 규칙

1. `90`의 사례 결과는 참고 자료일 뿐, 자신의 프로젝트 baseline이 아니다.
2. 적용하고 싶은 패턴은 자신의 `01. 요구사항 수집·리서치`에서 다시 문제·사용자·근거·제약과 연결한다.
3. 실제 업무에서 채택할 기능은 PRD의 `DECISION`과 acceptance criteria로 새로 승인한다.
4. 관찰일 이후 서비스가 바뀌었거나 근거가 오래되면 삭제하지 말고 `superseded`로 표시한다.
5. 시각 공유가 필요할 때 HTML·draw.io는 보조 자료로 만들 수 있지만, 기준 원문은 Markdown으로 유지한다.

## 7. 착수·완료 체크

### 착수 전

- [ ] URL과 관찰 목적이 정해졌다.
- [ ] 공개 접근·이용 조건·자동 수집 가능성을 확인했다.
- [ ] 제외 범위와 중단 조건을 기록했다.
- [ ] 관찰 결과를 복제가 아닌 추상화로 다룰 기준을 합의했다.

### 사례 문서 발행 전

- [ ] 각 `FACT`에 URL·관찰일·화면/경로 근거가 있다.
- [ ] `ASSUMPTION`, `DECISION`, `OPEN`이 구분돼 있다.
- [ ] 원문 카피·브랜드 asset·소스코드·개인정보가 포함되지 않았다.
- [ ] 현업 적용 포인트가 해당 서비스의 복제가 아니라 문제·사용자·제약의 재해석으로 작성됐다.
- [ ] `90` 문서가 `01~07`의 accepted baseline이 아님을 명시했다.

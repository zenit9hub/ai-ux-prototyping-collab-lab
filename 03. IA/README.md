# 03. IA

## 이 단계의 역할

여기는 PRD의 요구사항을 **정보 구조, 콘텐츠 모델, 사용자 과업 흐름, 상태와 예외 규칙**으로 번역하는 단계다. IA는 메뉴 목록만 만드는 문서가 아니며, 사용자가 어떤 정보를 어떤 순서로 보고 어떤 상태에서 어떤 행동을 할 수 있는지 정한다.

## 입력 기준

- `02. PRD`의 accepted PRD와 요구사항 traceability
- 우선순위, 수용 기준, 비범위, OPEN 목록
- 공통 제약: 역할·권한, 접근성, 기술·운영 조건

## 핵심 산출물

| 파일 예시 | 내용 | 다음 수신자 |
| --- | --- | --- |
| `<CASE-ID> — IA.md` | 메뉴/화면 구조, 콘텐츠 모델, screen ID | UX / UI Owner |
| `<CASE-ID> — User Flow and State Model.md` | task flow, 정상·빈·오류·권한·실패 상태, 전이 규칙 | UX / UI Owner |
| `<CASE-ID> — IA Flow.drawio` 또는 `.html` | 협업·설명용 시각 흐름, 기준 Markdown 링크 포함 | UX / UI Owner |

상세 형식은 [`01. 단계 산출물 템플릿`](<../00. 운영·공통 기준/templates/01. 단계 산출물 템플릿.md>)을 사용한다.

## IA가 답해야 할 질문

1. 요구사항 `R-##`은 어느 화면·콘텐츠·상태에 반영되는가?
2. 사용자는 어디에서 시작해 어떤 정보를 확인하고 다음 행동을 선택하는가?
3. 데이터가 비어 있거나, 실패하거나, 권한이 없으면 무엇을 보게 되는가?
4. 화면 간 이동 후 이전 맥락·필터·선택은 어떻게 유지되는가?
5. 다음 Wireframe이 해석 없이 사용할 screen/flow/state ID는 무엇인가?

## 하지 않는 일

- 색상, 타이포그래피, 시각 스타일을 최종 결정하지 않는다.
- state와 exception을 생략한 정상 흐름만 그리지 않는다.
- PRD의 요구사항을 화면에 억지로 1:1 배치해 사용자 과업을 왜곡하지 않는다.

## 품질 게이트: IA → Wireframe

- [ ] 핵심 `R-##` 요구사항이 `SCR-##`, `FLOW-##`, `STATE-##`에 매핑된다.
- [ ] 메뉴·화면·콘텐츠 모델·핵심 과업의 관계가 명확하다.
- [ ] 정상·빈·오류·권한·실패 상태가 필요한 범위에서 표현된다.
- [ ] 용어와 상태명이 PRD와 충돌하지 않는다.
- [ ] UX / UI Owner가 화면 구조와 행동 annotation을 시작할 수 있다.

## 다음 인수인계

`04. Wireframe`에는 accepted IA, user flow/state model, screen ID 목록, 필수 데이터 필드, 상태별 UX 원칙을 전달한다. 시각적 표현이 아닌 정보 우선순위와 행동 규칙이 핵심 전달물이다.

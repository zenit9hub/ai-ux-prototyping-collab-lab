# 협업 및 변경 기여 가이드

## 기본 원칙

이 저장소는 단계별 산출물이 다음 단계의 입력이 되는 협업 구조를 연습한다. 따라서 파일을 추가하는 것보다 **승인된 입력·명확한 결정·전달 가능한 출력**을 우선한다.

## 브랜치와 Pull Request

| 항목 | 규칙 |
| --- | --- |
| 기준 브랜치 | `main`에는 accepted baseline만 반영 |
| 작업 브랜치 | `work/<case-id>/<NN>-<stage-name>` 예: `work/aiot-platform/03-ia` |
| PR 범위 | 원칙적으로 한 사례의 한 단계 산출물 또는 하나의 변경 요청 |
| 병합 방식 | squash merge 권장 |
| 리뷰어 | 다음 단계 수신자 또는 해당 역할 reviewer |

PR은 Draft로 시작할 수 있다. 품질 게이트를 통과하고 reviewer가 수신 가능한 상태라고 판단한 뒤에만 병합한다.

## 파일명과 사례 식별

- 사례가 하나면 단계 폴더에 canonical 문서를 직접 둔다.
- 여러 팀이 같은 사례를 다루면 깊은 팀 폴더 대신 `T01_`, `T02_` 같은 파일명 prefix를 사용한다.
- 여러 사례가 동시에 진행될 때만 각 단계 폴더 아래에 동일한 `case-id` 디렉터리를 만든다.
- 기준 원문은 `CASE-01 PRD.md`처럼 역할과 사례가 드러나는 이름을 사용한다.

## 문서 상태

`draft` → `review` → `accepted` → `superseded`

`accepted`만 downstream baseline이 될 수 있다. 상위 산출물을 고칠 때는 `CR-###` 변경 요청을 만들고 영향을 받는 단계와 reviewer를 기록한다.

## AI 기여 기록

AI가 만든 초안이나 변형안을 채택할 때는 다음을 문서 또는 PR에 기록한다.

- 사용 도구와 모델/버전(알 수 있는 범위에서)
- 입력한 기준 문서와 제약
- 사람이 검토·수정·승인한 범위
- 남아 있는 `ASSUMPTION`과 `OPEN`

## 디자인 도구와 대형 파일

- Figma/Claude Design의 canonical 기준은 05 단계의 manifest에서 관리한다.
- Git에는 디자인 URL, 페이지/프레임 ID, 동결일, 버전, 필요한 export 목록을 기록한다.
- `.fig`, `.sketch`, 원본 영상, 내보낸 대용량 binary는 별도 승인 없이 넣지 않는다.
- 실제 프론트엔드 코드는 기본적으로 별도 개발 저장소에서 관리한다.
- 07. 배포·검증 (GitHub Pages)의 site는 교육·협업 검증용 정적 source다. main에 병합된 해당 폴더의 변경은 GitHub Pages로 자동 배포된다.
- 배포 화면에서 발견한 차이는 해당 Issue 또는 CR을 통해 상위 기준 문서로 연결한다.

## PR 전 최소 점검

- [ ] 직전 accepted baseline을 링크했다.
- [ ] 현재 단계의 입력·출력·완료 기준을 충족했다.
- [ ] 사실·가정·결정·미확정이 분리되어 있다.
- [ ] 다음 단계 수신자가 사용할 핵심 입력을 적었다.
- [ ] 상대 링크와 메타데이터 검증을 통과했다.

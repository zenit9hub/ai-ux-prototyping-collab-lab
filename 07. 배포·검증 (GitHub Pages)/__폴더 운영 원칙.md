# 07. 배포·검증 (GitHub Pages) · 폴더 운영 원칙

## 목적

기획·디자인·개발 전달의 결과를 누구나 같은 URL에서 확인하고, 발견한 차이를 원래 의사결정 단계로 되돌릴 수 있게 한다.

## 포함 범위

- 배포되는 정적 웹 소스
- GitHub Pages artifact와 연결되는 최소 배포 설정
- 검증용 mock data와 화면 상태
- 배포 결과를 확인하는 README와 연결 정보

## 포함하지 않는 것

- Figma 원본 또는 대형 디자인 export
- 실제 운영 서비스의 비밀값, 고객 데이터, API key
- 프로덕션 백엔드·배포 인프라·운영 데이터베이스
- 이전 결과물의 무분별한 복사본

## 인수인계 규칙

1. 06의 accepted handoff를 기준으로 site를 갱신한다.
2. site 변경은 main에 병합된 뒤 GitHub Pages에서 자동 배포된다.
3. 배포 화면에서 발견한 차이는 Issue 또는 CR로 기록한다.
4. 범위·구조·디자인 기준의 변경은 02~05 중 가장 이른 영향 단계에서 다시 승인한다.

## 작성 원칙

- 정적 웹은 index.html, CSS, JavaScript, mock data를 분리한다.
- 상대 경로만 사용해 GitHub Pages project URL에서도 동작하게 한다.
- mock data는 실제 운영 데이터처럼 오해되지 않게 코드와 화면에서 표시한다.
- 시각적 polish보다 기준 추적성, 상태 표현, 구현 전달 가능성을 우선한다.

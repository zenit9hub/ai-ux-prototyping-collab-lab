# 01. 요구사항 수집·리서치

## 이 단계의 역할

여기는 솔루션을 고르는 곳이 아니라, **문제·사용자·상황·제약의 근거를 수집하고 요구사항 후보를 분리하는 단계**다. 강의에서는 리서치 역할을 연습하고, 현업에서는 Researcher 또는 PO가 이 책임을 맡는다.

## 입력 기준

- `00. 운영·공통 기준`의 공통 context, 하네스, 품질 게이트
- 과제 브리프 또는 사례 설명
- 출처가 확인된 기존 자료, 인터뷰 메모, 관찰, 데이터

입력 자료가 충분하지 않으면 부족한 근거를 `OPEN`으로 남긴다. AI가 빈 부분을 실제 리서치처럼 채우지 않는다.

## 핵심 산출물

| 파일 예시 | 내용 | 다음 수신자 |
| --- | --- | --- |
| `<CASE-ID> — Evidence Map.md` | 출처·날짜·관찰·신뢰도·시사점 | Product Decision Owner |
| `<CASE-ID> — 요구사항 후보 및 질문.md` | FACT, ASSUMPTION, 리스크, OPEN, 요구사항 후보 | Product Decision Owner |
| `<CASE-ID> — 리서치 시각 요약.html` | 회의·공유용 보조 시각 자료, 기준 Markdown 링크 포함 | Product Decision Owner |

상세 형식은 [`01. 단계 산출물 템플릿`](<../00. 운영·공통 기준/templates/01. 단계 산출물 템플릿.md>)을 사용한다.

## 하지 않는 일

- PRD의 최종 우선순위나 범위를 확정하지 않는다.
- IA, 화면, Hi-Fi를 미리 결정하지 않는다.
- 출처 없는 페르소나, 고객 발언, 수치를 사실로 기록하지 않는다.

## 품질 게이트: Research → PRD

- [ ] 근거마다 출처·날짜·맥락이 있다.
- [ ] FACT와 ASSUMPTION, OPEN이 분리되어 있다.
- [ ] 문제·사용자·상황·제약이 구분되어 있다.
- [ ] 요구사항 후보가 단순 기능 목록이 아니라 해결하려는 문제와 연결된다.
- [ ] Product Decision Owner가 우선순위와 범위를 결정할 수 있다.

## 다음 인수인계
